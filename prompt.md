# Sprint 6.11 — Bruk Project Orchestrator for sammensatte oppgaver

> **Les CLAUDE.md. Les HELE denne filen før du endrer noe.**

---

## Problemet

Når bruker sier "Lag et repo som heter X og legg til en index.html":

**Nå:** AI kaller `create_task` to ganger → to frittstående tasks i `tasks`-tabellen → ingen kobling mellom dem → `autoInitRepo` lager ENDA en "Initialiser repo"-task → duplikater overalt → avhengigheter fungerer ikke → andre task starter aldri.

**Ønsket:** AI kaller `create_task` ÉN gang med tittelen "Lag repo X med index.html". Agenten dekomponerer dette INTERNT til sub-steps (opprett repo → lag fil → push). Brukeren ser ÉN task som rapporterer fremgang.

---

## Løsningen — Én task, interne steg

Ikke bruk `project_plans`/`project_tasks` for enkle oppgaver (det er for store prosjekter). I stedet: la AI-en lage ÉN task, og la agenten håndtere kompleksiteten internt.

### Endring 1 — AI system prompt (`ai/ai.ts`)

Oppdater `getToolInstructions()`. Fjern all logikk om at AI skal lage flere tasks med `dependsOn`. AI skal lage ÉN task per brukerforespørsel:

```
VIKTIGE REGLER FOR OPPGAVER:
1. Lag ALLTID kun ÉN oppgave per brukerforespørsel
2. Beskriv ALT brukeren ber om i oppgavens title og description
3. ALDRI lag flere tasks for å dekke én forespørsel
4. Agenten håndterer dekomponering internt (repo-opprettelse, filskriving, osv)

EKSEMPLER:
- "Lag repo X med index.html" → ÉN task: "Lag repo X med index.html fil"
- "Fiks bug Y og skriv tester" → ÉN task: "Fiks bug Y og skriv tester"
- "Bygg en komplett TODO-app" → ÉN task: "Bygg TODO-app med CRUD endpoints"

ALDRI GJØR DETTE:
- Lag separate tasks for "Opprett repo" og "Lag fil" — dette er ÉN oppgave
- Bruk dependsOn — dette er kun for orchestrator-modus
```

### Endring 2 — Fjern `dependsOn` fra `create_task` tool schema (`ai/ai.ts`)

I tool-definisjonen for `create_task`, fjern `dependsOn` fra schema:

```typescript
// FJERN dette fra create_task tool schema:
// dependsOn: { type: "array", items: { type: "string" }, description: "..." }
```

AI skal ikke tenke på avhengigheter i det hele tatt.

### Endring 3 — Auto-start etter create_task (`ai/ai.ts`)

Når AI kaller `create_task` og deretter `start_task`, er det vanlige flyten. Men vi må sikre at AI ikke lager FLERE tasks. 

I `executeToolCall("create_task")`:

### Endring 4 — Agenten håndterer repo-opprettelse som internt steg (`agent/agent.ts`)

`autoInitRepo` gjør allerede dette — den oppretter repo og initialiserer det som en del av agentens flyt. Det som trengs er at `autoInitRepo` IKKE lager en separat task i tasks-tabellen.

I `agent/helpers.ts`, `autoInitRepo()`:

Finn der den kaller `tasks.createTask()` for "Initialiser repo: X". **FJERN dette.** `autoInitRepo` skal gjøre arbeidet direkte uten å opprette en task for det:

```typescript
export async function autoInitRepo(ctx: AgentExecutionContext): Promise<void> {
  // ... ensureRepoExists (opprett repo på GitHub) ...
  
  // FJERN: const initTask = await tasks.createTask({ title: `Initialiser repo: ${repoName}`, ... });
  // FJERN: await tasks.updateTaskStatus({ id: initTask.id, status: "in_progress" });
  
  // I stedet: bare kjør init-arbeidet direkte
  await report(ctx, "Initialiserer repo med grunnfiler...", "working");
  
  const initFiles = [
    { path: "README.md", content: `# ${repoName}\n\nCreated by TheFold\n` },
    { path: ".gitignore", content: "node_modules/\ndist/\n.env\n" },
  ];
  
  // Push init-filer via createPR direkte
  try {
    await github.createPR({
      owner: ctx.repoOwner,
      repo: ctx.repoName,
      title: `init: Initialize ${repoName}`,
      body: "Auto-generated by TheFold",
      branch: `thefold/init-${Date.now()}`,
      files: initFiles,
    });
    await report(ctx, "Repo initialisert med README og .gitignore", "working");
  } catch (e) {
    // Hvis PR feiler (f.eks. tomt repo), prøv direkte push
    log.warn("autoInitRepo: createPR failed, continuing anyway", { 
      error: e instanceof Error ? e.message : String(e) 
    });
  }
  
  // FJERN: await tasks.updateTaskStatus({ id: initTask.id, status: "done" });
}
```

### Endring 5 — Rapport-meldinger i chatten

Agenten bør rapportere fremgang som steg, ikke som separate tasks. `report()` i agenten allerede gjør dette via `agent_status` meldinger.

Sjekk at agent-statusen viser tydelige steg:
- "Oppretter repo på GitHub..." 
- "Initialiserer repo med grunnfiler..."
- "Skriver index.html..."
- "Validerer kode..."
- "Lager PR..."

## Hva med store prosjekter?

For "Bygg en komplett TODO-app med frontend, backend, auth, database" — DER brukes `project_plans`/`project_tasks` via orchestrator. Men det er en SEPARAT flyt som aktiveres via `detectProjectRequest()` i `chat/detection.ts`.

Forskjellen:
- **Enkel oppgave** (repo + fil, bug fix, feature): ÉN task → agent håndterer internt
- **Stort prosjekt** (multi-service, mange filer): Project orchestrator → faser → mange tasks

Denne sprinten fikser kun den enkle oppgave-flyten.
